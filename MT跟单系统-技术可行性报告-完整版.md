## 1. 项目概述

### 1.1 项目背景

用户当前在 MetaTrader 4 平台上运行一个交易账户（账户 A），该账户通过外部信号源接收交易指令并自动执行。现需将账户 A 的交易操作同步复制到新账户（账户 B），并支持未来扩展到更多从账户（1 对多）。

### 1.2 项目目标

| 目标 | 描述 |
|---|---|
| **主目标** | 实现账户 A → 账户 B 的交易操作实时跟单同步 |
| **扩展目标** | 支持 1 个主账户对 N 个从账户的跟单架构 |
| **性能目标** | 端到端跟单延迟 ≤ 500ms |
| **可靠性目标** | 跟单成功率 ≥ 99.5% |

### 1.3 跟单范围

- 市价开仓/平仓
- 挂单（限价单 LIMIT、止损单 STOP）的创建与删除
- 持仓止损/止盈的修改
- 部分平仓
- 品种代码映射（不同经纪商之间的品种后缀差异）

## 3. 技术可行性

### 3.1 MT4 交互方案

**方案 A：ZeroMQ 事件推送（推荐）**

| 维度 | 评估 |
|---|---|
| **原理** | MQL4 EA 调用 `mql-zmq` DLL，通过 ZeroMQ PUB socket 将 `OnTrade()` 事件实时推送到 Python |
| **延迟** | < 1ms（事件触发到推送） |
| **成熟度** | 高 — Darwinex DWX Connector 已验证的工业方案 |
| **依赖** | `libzmq.dll` + `mt4zmq.dll` 需部署到 MT4 目录，终端需允许 DLL 导入 |
| **许可证** | MIT 开源 |
| **风险** | MetaQuotes 未来的终端更新可能限制 DLL 导入 |

**方案 B：文件管道轮询**

| 维度 | 评估 |
|---|---|
| **原理** | EA 将交易信息写入共享文件，Python 定时轮询读取 |
| **延迟** | 取决于轮询间隔和磁盘 IO，通常 50-200ms |
| **成熟度** | 高 — 多数商业跟单 EA 采用此方案 |
| **依赖** | 无额外 DLL 依赖 |
| **风险** | 磁盘 IO 压力、文件锁冲突 |

### 3.2 MT5 交互方案

**使用官方 `MetaTrader5` Python 库（MetaQuotes 官方）**

| 维度 | 评估 |
|---|---|
| **原理** | Python 通过官方库直接调用 MT5 终端的 `positions_get()` / `order_send()` 等函数 |
| **延迟** | 取决于轮询间隔（推荐 200-300ms） |
| **成熟度** | 高 — MetaQuotes 官方维护 |
| **依赖** | 仅需 `pip install MetaTrader5`，无 DLL 依赖 |
| **优势** | 官方支持、API 稳定、部署简单 |

### 3.3 MetaTrader Signals（内置信号功能）可行性评估

| 维度 | 评估 | 结论 |
|---|---|---|
| **延迟** | 云端中继 + 终端轮询，5-30 秒 | ❌ 不可接受 |
| **级联支持** | 一个账户不能同时是订阅者+提供者 | ❌ 不适用 |
| **挂单同步** | 不支持 | ❌ 功能缺失 |
| **自定义比例** | 仅自动按余额比例 | ❌ 不可控 |
| **结论** | **不适用于本项目的级联跟单场景** | — |

### 3.4 技术选型总结

| 层级 | 选型 | 理由 |
|---|---|---|
| **语言** | Python 3.12+ | 生态丰富，MT5 官方库支持 |
| **MT4 交互** | ZeroMQ 桥接（MQL4 EA + `pyzmq`） | 最低延迟（< 1ms 触发），工业验证 |
| **MT5 交互** | `MetaTrader5` 官方库 | 稳定、无额外依赖 |
| **异步框架** | `asyncio` | 同时管理多终端连接 |
| **数据库** | PostgreSQL 16 | 订单映射与审计日志 |
| **缓存** | Redis 7 | 快照缓存、分布式锁、消息队列 |
| **部署** | Docker + Windows VPS | 核心容器化 + MT 终端独立运行 |

### 4.2 数据流

```
账户 A 产生交易 → EA OnTrade() 触发 → ZeroMQ PUB 推送 JSON 信号 (本地 <1ms)
    → Python 信号检测器(去重) → 查询跟单配置(Redis 缓存)
    → 风控检查 → 比例计算 → 精度截断
    → MT4 下单执行(ZMQ 命令) 或 MT5 下单执行(mt5.order_send)
    → 写入订单映射(PostgreSQL) → 审计日志
```

### 4.3 关键架构决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 信号捕获方式 | ZeroMQ 事件推送（MT4） | 延迟最低，专为高频交易通信设计 |
| 信号通信通道 | localhost（同机） | 延迟 < 1ms vs TCP 跨机 5-50ms |
| 进度保证 | Redis Streams 消息队列 | 不丢消息，支持消费者组 ACK |
| 去重策略 | Redis SETNX + ticket+timestamp 联合键 | 防止 `OnTrade()` 重复触发导致的重复下单 |

## 5. 核心技术方案

### 5.1 信号检测：快照差量对比法

```
算法流程（单次轮询）：
  1. 调用 positions_get() 获取当前持仓快照 {ticket → Position}
  2. 从 Redis 读取上一轮快照
  3. 差量对比：
     · 新 ticket → OPEN 信号（开仓）
     · 消失的 ticket → CLOSE 信号（平仓）
     · SL/TP 变化 → MODIFY 信号（修改）
  4. 更新快照到 Redis
  5. 发布事件到 Redis Streams
```

### 5.2 跟单执行：比例计算与精度截断

```
跟单手数 = truncate(master_volume × copy_ratio, volume_step)

示例：
  主账户买入 0.10 手 EURUSD
  跟单比例 50%
  原始 = 0.10 × 0.50 = 0.05
  如果 EURUSD volume_step = 0.01 → 0.05 ✅
  如果 volume_step = 0.1 → truncate(0.05, 0.1) = 0.00 → 跳过（小于最小手数）
```

### 5.3 循环触发防护

三层防护机制：

| 层级 | 方法 | 说明 |
|---|---|---|
| L1 - Comment 标记 | 从账户下单时自动附加 `[COPY-LEAD-A]` | 信号检测器识别并忽略自身产生的订单 |
| L2 - Ticket 白名单 | Redis Set 维护本系统发出订单的 ticket | 查询快照时排除已知 ticket |
| L3 - 来源校验 | 检测器记录每个订单的来源 | 系统发出的不触发 |

### 5.4 断连补偿对账

```
补偿流程：
  1. WebSocket/ZMQ 断连检测（心跳超时 > 30s）
  2. 重连后立即执行全量对账：
     a. 拉取主账户全量持仓和挂单
     b. 计算每个从账户的预期持仓
     c. 拉取从账户实际持仓
     d. 差量修复：补单 / 平多余仓位 / 调整 SL-TP
  3. 写入对账日志和异常告警
```

### 5.5 风控体系

```
风控检查链（每次下单前执行）：
  ┌─ 余额检查：账户余额 × 保证金率 > 所需保证金
  ├─ 仓位上限：当前品种持仓 + 新开仓 ≤ max_position
  ├─ 数量上限：当前总持仓数 < max_positions
  ├─ 日亏损检查：当日已实现亏损 < max_daily_loss
  ├─ 回撤检查：当前回撤 < max_drawdown
  └─ 下单间隔：距上次下单 ≥ 500ms
```

## 7. 风险分析

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| **MetaQuotes 限制 DLL 导入** | 低 | 高 | 准备文件管道方案作为备选；MT5 不受影响 |
| **经纪商限制高频交易** | 中 | 中 | 内置 ≥ 500ms 下单间隔控制，可配置 |
| **MT4 终端崩溃** | 中 | 高 | 进程监控 + 自动重启 + 恢复后全量对账 |
| **ZeroMQ 连接中断** | 低 | 中 | 心跳检测 + 自动重连 + 指数退避 |
| **主从品种代码不一致** | 中 | 中 | 维护品种映射表，支持正则匹配 |
| **循环跟单** | 低 | 高 | 三层防护机制 |
| **从账户余额不足** | 中 | 低 | 自动降级：跳过或按最大可用比例调整 |
| **网络分区（VPS 断网）** | 低 | 高 | 恢复后全量对账补偿 |

## 9. 结论与建议

### 9.1 可行结论

**项目完全可行。** MetaTrader 4/5 的 API 生态成熟，ZeroMQ 桥接方案经过工业验证，本方案在技术上没有不可逾越的障碍。

### 9.2 核心建议

| 建议 | 优先级 |
|---|---|
| 采用 **MT4 + ZeroMQ 推送方案** 作为主方案，延迟 50-200ms | P0 |
| 全部组件部署在 **同一台 Windows VPS** 上，local 通信消除网络延迟 | P0 |
| 先实现 1 对 1（账户 A → 账户 B），验证稳定后再扩展 | P0 |
| 准备文件管道方案作为 **MT4 DLL 限制时的备选** | P1 |
| 管理后台按需开发，初期可通过配置文件 + 日志满足管理需求 | P2 |
| 保留未来扩展 **MT5 方案** 的能力（通过统一抽象层） | P2 |
